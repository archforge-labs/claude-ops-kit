# /debug

症状から根本原因まで、構造化されたデバッグフローを実行する。

## 動作

1. **症状の言語化**
   - 「何が起きているか」「何が期待されるか」を明確に区別する
   - エラーメッセージ・スタックトレース・再現手順を収集

2. **仮説の生成**
   - 考えられる原因を可能性の高い順にリストアップ（最大 5 個）
   - 各仮説に「確認方法」を紐付ける

3. **最小再現の設計**
   - 変数を最小化して問題を分離する
   - 「これが原因ならこうなるはず」という予測を立ててから確認する

4. **仮説の検証ループ**
   - 最も可能性の高い仮説から検証
   - 外れたら消去して次へ（闇雲に触らない）

5. **修正と記録**
   - 修正後に再現手順で確認
   - 根本原因がアーキテクチャに関わる場合は `/adr` で記録

## プロセス

```
症状の整理
  → 仮説リスト（可能性順）
    → 最小再現
      → 検証ループ（外れたら次の仮説）
        → 修正
          → 確認
            → 必要なら /adr で記録
```

## 使用例

```
/debug
```
→ 現在直面しているバグをガイドに沿って分析する

```
/debug TypeError: Cannot read properties of undefined
```
→ エラーメッセージを起点に仮説を生成する

## デバッグの原則

- **1 変数ずつ変える**: 複数の変更を同時に試さない。何が効いたかわからなくなる
- **仮説を先に書く**: 「こうなるはず」を予測してから確認する。予測と結果のズレが情報になる
- **ログを信じる、記憶を疑う**: 「たぶんこうなってるはず」より `console.log` / `print` を使う
- **再現しなければ直ってない**: 「たぶん直った」は直っていない

## 関連

- `/adr` — 根本原因がアーキテクチャ上の問題だった場合に記録
- `/standup` — ブロッカーとして報告する場合
